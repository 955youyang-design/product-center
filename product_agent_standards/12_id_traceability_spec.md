## 1. 文档目标

本文件定义稳定 ID 的命名规则、废弃规则，以及需求追踪矩阵的最小要求。

目标是保证文档在多轮迭代后，仍能知道“这个节点现在在哪、以前在哪、对应哪些文档和测试”。

## 2. 稳定 ID 原则

- 名称可以调整，ID 原则上不改
- ID 一旦进入正式基线，不应重复使用
- 节点下线时应废弃，不应删除后复用同一个 ID
- 大版本重构时，优先保留旧 ID 的替代关系，而不是直接丢失历史

## 3. 建议 ID 类型

建议至少覆盖以下类型：

| 节点类型 | 建议字段 | 示例 |
|---|---|---|
| 功能 | `feature_id` | `FEAT-product-001` |
| 页面组 | `page_group_id` | `PG-product-001` |
| 页面 | `page_id` | `PAGE-product-001` |
| 菜单 | `menu_key` | `MENU-product-001` |
| 路由 | `route_key` | `ROUTE-product-001` |
| 机制 | `mechanism_id` | `MECH-product-001` |
| 对象 | `object_id` | `OBJ-product-001` |
| 接口 | `api_id` | `API-product-001` |
| 测试用例 | `test_case_id` | `TC-product-001` |
| 问题 | `issue_id` | `ISSUE-product-001` |
| 假设 | `assumption_id` | `ASSUMP-product-001` |
| 文档 | `doc_id` | `DOC-product-001` |

## 4. 建议命名规则

建议采用：

`前缀-领域英文标识-三位序号`

例如：

- `FEAT-catalog-001`
- `PAGE-catalog-012`
- `API-inventory-003`

命名要求：

- 前缀固定
- 领域标识使用英文小写
- 序号建议从 `001` 开始
- 不建议把中文名直接写进 ID
- 中文名称、菜单文案、页面标题应视为展示层，不应直接承担稳定追踪职责

## 5. 废弃与替代规则

如果节点被替换或下线，建议至少记录：

- `current_status`
- `supersedes`
- `deprecated_reason`

处理规则：

- 旧 ID 保留在历史记录中
- 新 ID 明确写出替代关系
- 追踪矩阵中保留旧链路，避免断链

## 6. 需求追踪矩阵的适用场景

以下场景建议建立需求追踪矩阵：

- 多业务线
- 多终端
- 多 Agent 协同
- 页面数较多
- 有长期迭代计划

## 7. 追踪矩阵最小字段

建议至少包含以下字段：

| feature_id | page_group_id | page_id | menu_key | route_key | mechanism_id | object_id | api_id | test_case_id | doc_ref | current_status | notes |
|---|---|---|---|---|---|---|---|---|---|---|---|

字段说明：

- `doc_ref`：当前节点主要承接在哪份文档
- `current_status`：当前范围 / 扩展预留 / 已下线 / 已归档

## 8. 最小追踪要求

至少应满足：

- 每个 `feature_id` 至少能追到一个端产品层章节
- 每个 `page_group_id` 至少能追到一个页面组文档
- 每个 `page_id` 至少能追到一个页面文档或页面组章节
- 每个 `mechanism_id` 至少能追到一个专题机制文档
- 每个 `object_id` 或 `api_id` 至少能追到一个数据与接口层文档
- 每个关键 `feature_id` 最终能追到至少一个 `test_case_id`

## 9. 输出位置建议

项目实例中，需求追踪矩阵建议放在：

`docs/00-总览层/08-需求追踪矩阵.md`

如果项目规模更大，也可以拆成：

- `08-需求追踪矩阵.md`
- `09-ID映射与废弃记录.md`

## 10. 迭代更新规则

每次范围、页面、机制、对象、接口或测试变化时，应同步更新：

- 对应基线表中的 ID
- 追踪矩阵中的关联关系
- 替代关系和废弃状态

如果文档已回写但追踪矩阵未回写，视为追踪链未完成。

## 11. 完成标准

一套合格的 ID 与追踪矩阵规范落地后，至少应满足：

- 关键节点在多轮迭代后仍能追溯
- 不会因为名称变化导致上下游断链
- 测试和实现都能回挂到需求节点
