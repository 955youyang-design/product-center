# 产品 Agent 统一要求文档

| 字段 | 内容 |
|---|---|

| 文档类型 | 统一要求文档 |
| 适用范围 | 从 0 到 1 产品设计、需求迭代、文档治理 |
| 适用角色 | 产品经理、产品 Agent、交互设计、研发、测试 |

## 1. 文档目标

本文件定义一套统一要求，使产品 Agent 能够：

1. 通过与产品的对话收敛需求，而不是直接跳到页面细节。
2. 从 0 到 1 输出一套可直接支撑设计、研发、测试的产品文档。
3. 在需求持续变化时，按统一规则维护文档，而不是反复重写。

## 2. 核心原则

### 2.1 先对话澄清，再进入文档生产

产品 Agent 的起点不是写页面文档，而是做结构化澄清。

固定顺序如下：

1. 明确业务目标
2. 明确角色、端和渠道
3. 明确版本范围与范围外内容
4. 建立输入基线
5. 生成分层文档
6. 管理后续迭代

### 2.2 先基线，后分层文档

任何项目在正式输出实例文档前，必须先有三份输入基线：

1. 功能清单表
2. 页面树表
3. 功能承载图

这三份基线是上游输入，`docs/` 分层文档是下游派生文档。

### 2.3 页面、功能动作、规则能力分开治理

- 页面树表只承接页面节点
- 功能动作不冒充页面
- 规则能力不冒充页面
- 非页面节点必须进入功能清单表，并在功能承载图中表达承接关系

### 2.4 任何变化先改基线，再改派生文档

不得绕过输入基线，直接修改下游文档。

固定回写顺序如下：

1. 更新功能清单表
2. 更新页面树表
3. 更新功能承载图
4. 做影响分析
5. 更新受影响的分层文档

### 2.5 所有节点必须有稳定 ID

至少要求：

- 功能节点使用 `feature_id`
- 页面节点使用 `page_id`
- 页面组节点使用 `page_group_id`

名称可以调整，ID 不应频繁变化。

稳定 ID 的命名规则和废弃规则，以 `12_id_traceability_spec.md` 为准。

页面、菜单、路由、文档标题与文件路径的解耦规则，以 `16_navigation_naming_resilience_spec.md` 为准。

### 2.6 优先级统一使用数字，从 0 开始

优先级规则固定为：

- `0` 表示最高优先级
- 数字越小，优先级越高
- 不再使用 `P0 / P1 / P2` 口径

### 2.7 文档阶段的疑问与不确定项必须显式整理

在文档阶段，只要出现疑问、不确定项或多种可行解释，产品 Agent 都不得静默处理。

必须执行以下动作：

1. 进入问题清单
2. 标记影响范围和阻塞程度
3. 指定待确认的人类角色
4. 给出默认处理方式
5. 在人类确认后回写结论

如果问题阻塞范围判断、页面结构、机制口径、对象定义或验收口径，不得直接进入下一阶段正式文档输出。

### 2.8 正式文档应优先使用统一模板骨架

产品 Agent 不应每次重新发明章节结构。

正式文档输出时，应优先复用统一模板库，确保：

- 元信息字段一致
- 章节顺序一致
- 缺项容易发现
- 多 Agent 输出可汇总

模板库以 `11_template_library.md` 为准。

### 2.9 中大型项目应建立追踪矩阵

当项目存在以下任一情况时，建议输出需求追踪矩阵：

- 多业务线
- 多终端
- 多 Agent 协同
- 页面数较多
- 需要长期迭代维护

追踪矩阵规则以 `12_id_traceability_spec.md` 为准。

### 2.10 正式交付前应经过人类评审签收

产品 Agent 的自检完成，不等于项目文档已经正式可交付。

在正式交付前，应至少完成：

- 人类评审
- 评审意见归并
- 返修回写
- 签收记录
- 交付门槛判断

详细规则以 `13_review_signoff_spec.md` 为准。

### 2.11 项目实例应维护当前有效版本索引

当项目文档逐步增多、拆分、归档后，必须有一份“当前有效版本清单”作为定位入口。

详细规则以 `14_delivery_index_spec.md` 为准。

### 2.12 展示名可变，但稳定键不可乱变

为了支持中文文档名称、菜单名称、页面名称频繁调整而不破坏链路，必须区分：

- 展示名
- 稳定键
- 文件路径
- 路由键

展示名允许调整，稳定键和追踪关系原则上保持不变。

## 3. 产品 Agent 必须具备的能力

产品 Agent 至少必须能完成以下工作：

- 通过对话收集需求并识别信息缺口
- 对不阻塞结构的缺口做有限假设，并明确标记
- 在信息不完整时先收敛基线，再展开细节
- 输出结构化产品文档，而不是零散说明
- 根据变化范围定位受影响文档，而不是整体重写

## 4. 从 0 到 1 的标准工作流

### 4.1 对话收敛阶段

产品 Agent 需要先收敛以下问题：

- 做什么产品
- 解决什么问题
- 谁来使用
- 哪些端、渠道、角色在当前范围内
- 当前版本做什么，不做什么
- 是否存在多条业务线
- 哪些业务线属于当前版本

### 4.2 基线构建阶段

在没有形成三份基线前，不进入正式页面规格和对象接口定义。

必须先完成：

1. 功能清单表
2. 页面树表
3. 功能承载图

### 4.3 分层文档阶段

项目实例文档必须按如下六层输出：

`总览 -> 端产品 -> 专题机制 -> 页面设计 -> 数据与接口 -> 验收与测试`

### 4.4 评审与补齐阶段

每一轮输出后，产品 Agent 必须显式给出：

- 已确认内容
- 假设内容
- 待确认问题
- 当前稳定文档
- 暂缓到后续迭代的内容

## 5. 必要输出物

一轮完整的从 0 到 1 输出，至少应包含：

1. 范围摘要
2. 假设清单
3. 待确认问题清单
4. 功能清单表
5. 页面树表
6. 功能承载图
7. `00-05` 分层文档
8. 本轮版本范围说明
9. 问题清单与人类确认记录（如存在疑问或不确定项）
10. 需求追踪矩阵（中大型项目建议产出）
11. 项目交付索引与当前有效版本清单

强制输出文档清单、触发条件和可合并规则，统一以 `07_required_output_manifest.md` 为准。

## 6. 分层文档最小要求

### 6.1 00-总览层

回答产品边界、角色协同、主流程、状态流转与术语口径。

### 6.2 01-端产品层

回答每个端承接什么功能、页面组和页面如何组织，并承接功能清单表、页面树表和功能承载图。

### 6.3 02-专题机制层

回答规则能力、跨页面机制、跨对象机制如何触发、流转、失败和收口。

### 6.4 03-页面设计层

回答页面目标、结构、布局、操作、字段、校验、状态与反馈。

### 6.5 04-数据与接口层

回答对象、字段、状态、接口、事件、日志与表设计建议。

### 6.6 05-验收与测试层

回答主流程、异常流程、权限流程和状态流转如何验收。

### 6.7 页面开发脚手架定义

当页面已经进入研发实施范围时，除页面规格外，还应补充页面开发脚手架定义。

页面开发脚手架定义的作用是：

- 把页面文档稳定映射为可实施的前端骨架
- 统一页面目录、路由、区块、状态、接口挂载方式
- 降低不同开发者或 Agent 对同一页面理解不一致的风险

详细规则以 `09_page_scaffold_spec.md` 为准。

### 6.8 模板库与追踪矩阵

为了保证输出可执行、可复核、可迭代，建议同时维护：

- 统一模板库
- 稳定 ID 规则
- 需求追踪矩阵

相关规则分别见：

- `11_template_library.md`
- `12_id_traceability_spec.md`

### 6.9 基线存放与交付索引

为了避免三份输入基线和项目实例文档散落，项目应同时明确：

- 基线存放位置与载体
- 当前有效文档索引

相关规则分别见：

- `14_delivery_index_spec.md`
- `15_baseline_storage_format_spec.md`

## 7. 流程图与时序图规则

整体业务流程文档必须满足以下要求：

- 至少有 1 张总览流程图
- 至少有 1 张时序图
- 如果存在多条业务线，必须按业务线拆分为多张时序图
- 时序图用于表达角色、页面、机制、对象、接口之间的先后关系
- 时序图不替代页面流程图，也不替代功能承载图

## 8. 迭代阶段要求

在进入迭代阶段后，产品 Agent 必须遵守以下顺序：

1. 先更新三份基线
2. 再做影响分析
3. 再更新受影响的分层文档
4. 最后输出变更记录和差异摘要

禁止的做法包括：

- 直接重写页面文档但不回写基线
- 直接新增页面但不更新页面树
- 直接改承接关系但不更新功能承载图

## 9. 完成标准

满足以下条件，才算一轮合格输出：

- 范围、角色、端和渠道边界清晰
- 功能清单表、页面树表、功能承载图三者一致
- 分层文档已形成上下游关系
- 假设与待确认问题被明确标记
- 范围变化后，可以快速判断需要回写哪些文档

进一步的可审计完成标准如下：

- 所有 `in scope` 的 `page_id` 都能映射到页面文档或页面组文档
- 所有 `rule_capability` 都能映射到专题机制文档
- 所有 `data_capability` 都能映射到对象、字段或接口文档
- 所有未决问题都已标记阻塞程度和默认处理方式
- 中大型项目中的关键节点都已进入追踪矩阵

## 10. 多 Agent 协同原则

如果采用多个产品经理 Agent 协同，必须满足以下原则：

- 主 Agent 负责范围收敛、任务拆分和最终汇总
- 子 Agent 只能在被分配的边界内产出，不得擅自扩范围
- 任何子 Agent 产出的页面、机制、对象定义，都必须回挂到统一基线
- 最终统一口径、命名和冲突合并责任在主 Agent

多 Agent 的详细协同规则，以 `08_multi_agent_collaboration.md` 为准。

## 11. 人类确认规则

产品 Agent 可以整理问题、提出候选方案、给出默认处理建议，但不能把人类确认替换掉。

以下内容原则上必须由人类确认：

- 当前版本范围边界
- 多个候选业务规则中的最终口径
- 会影响多个文档层级的术语和状态定义
- 会影响研发实施和验收结果的关键取舍

详细规则以 `10_issue_confirmation_spec.md` 为准。

## 12. 文档边界

本文件是通用标准，不写任何具体项目内容。

所有项目实例内容应在项目自己的 `docs/` 目录中完成。
