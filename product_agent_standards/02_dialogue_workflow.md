# 产品对话工作流规范

## 1. 文档目标

本文件定义产品 Agent 与产品对话时的标准流程，确保需求收集不是“想到哪写到哪”，而是能稳定沉淀为后续可落地的产品文档。

## 2. 对话的固定阶段

产品 Agent 应按以下阶段推进对话：

1. 明确业务目标
2. 明确用户、角色和使用场景
3. 明确端、渠道和业务线
4. 明确当前版本范围与范围外内容
5. 收敛三份输入基线
6. 输出分层文档
7. 进入评审与迭代回写

## 3. 每轮对话必须澄清的最小问题集

至少需要澄清以下问题：

- 这个产品要解决什么问题
- 谁是使用者，谁是管理者，谁是协同角色
- 当前版本支持哪些端和渠道
- 当前版本明确要做哪些能力
- 当前版本明确不做哪些能力
- 是否需要编辑、审核、发布、同步、导入、导出、日志、权限等能力
- 当前是否存在多条业务线
- 多条业务线里，哪些进入当前版本

## 4. 信息不完整时的处理规则

### 4.1 阻塞型缺口

如果缺失信息会直接影响结构判断，产品 Agent 应优先提问，不应自行补全。

典型阻塞型缺口包括：

- 当前版本支持哪些端或渠道
- 当前范围内有哪些主角色
- 某条业务线是否纳入本期
- 某个能力是页面、功能动作还是规则能力

### 4.2 非阻塞型缺口

如果缺失信息不会阻塞结构搭建，可以做有限假设，但必须显式标记。

典型非阻塞型缺口包括：

- 次级字段命名
- 个别页面文案
- 次要按钮顺序
- 暂未锁定的异常提示文案

### 4.3 明确禁止事项

产品 Agent 不得：

- 自行创造新的业务线
- 把未确认能力直接写入正式范围
- 用页面细节替代范围定义

## 5. 对话输出格式要求

每一轮结构化对话后，产品 Agent 应尽量按以下顺序反馈：

1. 已确认内容
2. 假设内容
3. 待确认问题
4. 当前范围边界
5. 已更新的基线项
6. 下一步将产出的文档

这样做的目的，是让产品在对话阶段就能看到“已经收敛了什么、还没收敛什么”。

## 6. 从对话到文档的门槛规则

产品 Agent 不得从对话直接跳到页面设计。

固定路径如下：

`对话澄清 -> 功能清单表 -> 页面树表 -> 功能承载图 -> 分层文档`

没有完成三份输入基线前，不进入正式页面规格、对象接口和测试用例阶段。

进入基线阶段前，至少必须确认以下内容：

- 业务目标
- 目标用户与核心角色
- 当前版本范围边界
- 当前支持的端、渠道
- 主业务线

以上任一项未确认时，不应进入正式基线输出。

以下内容允许带假设推进，但必须记录：

- 次级页面文案
- 非关键字段的默认值
- 次要交互提示
- 非主链路异常文案

## 7. 多轮对话下的推进规则

### 7.1 前期探索轮

目标是快速收敛范围，而不是快速把文档写厚。

此阶段允许：

- 多次修改功能清单表
- 多次调整页面树
- 多次刷新功能承载图

### 7.2 收口轮

当产品确认当前版本范围后，产品 Agent 应：

- 冻结当前基线版本
- 输出正式分层文档
- 标记范围外和延后项

### 7.3 迭代轮

如果产品在后续新增、删除或调整范围，产品 Agent 应：

1. 先回写基线
2. 再判断影响范围
3. 只更新受影响文档

## 8. 产品 Agent 的对话提示要求

在对话中，产品 Agent 应尽量做到：

- 提问简洁，一次只问能推动结构收敛的问题
- 对不清楚的项给出候选结构，帮助产品快速判断
- 不用模糊表达替代明确结构

推荐使用的结构化标签：

- `已确认`
- `待确认`
- `假设`
- `范围外`
- `下轮补充`

## 9. 假设清单与未决问题清单模板

假设清单建议至少包含以下字段：

| assumption_id | 假设内容 | 影响范围 | 默认处理方式 | 待谁确认 | 预计确认节点 | 当前状态 |
|---|---|---|---|---|---|---|

未决问题清单建议至少包含以下字段：

| issue_id | 问题内容 | 影响范围 | 阻塞程度 | 默认处理方式 | 待谁确认 | 预计确认节点 | 当前状态 |
|---|---|---|---|---|---|---|---|

阻塞程度建议至少区分：

- `阻塞基线`
- `阻塞页面设计`
- `阻塞研发实现`
- `不阻塞当前推进`

## 10. 文档阶段的问题整理规则

从进入基线输出开始，到分层文档完成之前，产品 Agent 应持续维护“问题清单”，而不是只在对话阶段临时记录。

建议至少归类为以下问题类型：

- `范围问题`
- `页面结构问题`
- `机制规则问题`
- `对象字段问题`
- `接口依赖问题`
- `验收口径问题`
- `术语命名问题`

每个问题至少要说明：

- 问题本身是什么
- 它影响哪些文档
- 当前默认处理方式是什么
- 如果不确认，会阻塞到哪一步

## 11. 人类确认要求

当问题属于以下情况时，必须发起人类确认：

- 会改变当前版本范围
- 会导致页面树或功能承载图出现两种以上合理结构
- 会影响规则能力、状态流转或关键对象口径
- 会影响验收是否通过
- 会导致不同 Agent 或不同角色理解不一致

建议在问题清单中增加以下确认字段：

| issue_id | 待确认角色 | 待确认人 | 确认方式 | 确认结论 | 确认时间 | 关闭状态 |
|---|---|---|---|---|---|---|

`确认方式` 建议至少支持：

- `口头确认`
- `文档批注确认`
- `会议确认`
- `消息确认`

## 12. 默认处理规则

在未收到人类确认前，产品 Agent 可以按默认处理方式继续推进，但必须满足：

- 默认处理不改变当前已确认范围
- 默认处理不会引入新的主业务线
- 默认处理已在问题清单中显式记录
- 默认处理结果在文档中可追溯

如果问题属于“阻塞基线”或“阻塞页面设计”，则不得仅靠默认处理进入正式交付状态。

## 13. 评审提示要求

当产品 Agent 完成一轮输出时，必须显式告知产品：

- 哪些内容已经稳定
- 哪些内容仍是临时假设
- 哪些问题如果不确认，会影响下一步文档
- 当前是否可以进入页面设计和对象接口定义

同时应附带：

- 当前未关闭问题数
- 其中阻塞问题数
- 已经完成的人类确认数
- 仍待确认的问题清单

## 14. 结果要求

一套合格的对话工作流，最终必须能把“口头需求”稳定转成“基线 + 文档”，而不是停留在讨论记录层面。
