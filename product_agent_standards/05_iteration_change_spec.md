## 1. 文档目标

本文件定义从首版输出之后，文档如何随着需求变化持续演进。

核心目标不是“把文档改完”，而是“按可追溯、可控、可复核的方式改完”。

## 2. 核心规则

任何变更发生后，都不得先改下游文档。

固定顺序如下：

1. 回写输入基线
2. 做影响分析
3. 更新受影响的分层文档
4. 记录本轮变更摘要

## 3. 常见变更类型

至少覆盖以下变更类型：

- 新增功能
- 删除功能
- 功能改名
- 页面新增或删除
- 页面树层级调整
- 承接关系调整
- 优先级调整
- 版本范围调整
- 对象字段调整
- 状态口径调整

## 4. 回写顺序

### 4.1 范围类变更

如果变化影响“做什么”，先改：

1. 功能清单表
2. 页面树表
3. 功能承载图

### 4.2 页面结构类变更

如果变化影响“页面如何组织”，先改：

1. 页面树表
2. 功能承载图
3. 页面设计层文档

### 4.3 机制与对象类变更

如果变化影响“规则如何跑”或“对象如何定义”，先改：

1. 功能承载图或对象定义
2. 专题机制层/数据与接口层
3. 页面设计层和测试层

## 5. 影响分析矩阵

| 变化类型 | 先改什么 | 重点影响层 |
|---|---|---|
| 新增/删除功能 | 功能清单表 | 00、01、03、05 |
| 新增/删除页面 | 页面树表 | 00、01、03、05 |
| 功能动作改承接位置 | 功能清单表、功能承载图 | 01、03、05 |
| 规则能力改承接位置 | 功能清单表、功能承载图 | 01、02、03、05 |
| 页面入口/出口变化 | 页面树表 | 01、03、05 |
| 对象字段变化 | 对象定义 | 04、05 |
| 状态口径变化 | 状态字典 | 00、02、04、05 |

## 6. 变更记录字段建议

建议在功能清单表和页面树表中增加以下字段，用于支撑后续迭代：

- `change_type`
- `change_reason`
- `changed_by`
- `changed_at`
- `effective_version`

如有条件，也可以补充：

- `change_ticket`
- `review_status`
- `affected_docs`
- `affected_trace_rows`

## 7. 锁口规则

### 7.1 未锁口阶段

- 允许持续修改三份基线
- 允许文档处于草案状态
- 重点目标是收敛结构，而不是追求一次性定稿

### 7.2 锁口阶段

- 当前版本范围冻结
- 分层文档进入正式输出状态
- 变更必须带影响分析

### 7.3 锁口后变更

锁口后任何新增、删除、调整，都必须满足：

1. 有变更原因
2. 有影响分析
3. 有定向回写
4. 有变更摘要

## 8. 版本维护建议

推荐采用以下口径：

- 大版本：`V1`、`V2`
- 小修订：`V1.1`、`V1.2`

使用规则：

- 范围调整但未上线，可记录为小修订
- 范围明显扩大或业务边界变化，可升级大版本
- 基线表和项目实例文档的版本口径应保持一致

补充建议如下：

- 当三份输入基线发生结构性变化时，优先提升 `baseline_version`
- 当仅为文档表达完善、不改变基线结构时，优先提升 `doc_version`
- 如果文档仍基于同一版基线，只做文档修订，不应伪装成新的基线版本

## 9. 迭代轮输出要求

每一轮迭代至少输出：

1. 已回写的功能清单表
2. 已回写的页面树表
3. 已回写的功能承载图
4. 受影响文档列表
5. 变更摘要
6. 待确认问题

如果本轮变更引入新的疑问或不确定项，还应补充：

7. 问题清单更新
8. 已完成的人类确认记录

如果项目已建立需求追踪矩阵，还应补充：

9. 追踪矩阵更新记录

如果项目已建立交付索引，还应补充：

10. 当前有效版本清单更新记录

## 10. 产品 Agent 的执行要求

在迭代阶段，产品 Agent 必须做到：

- 不重写无关文档
- 不擅自修改稳定 ID
- 保持跨文档命名一致
- 明确标记新增假设和待确认项
- 能解释本轮变化为什么影响这些文档，而不是另外一些文档

## 11. 文档生命周期状态

为了支撑长期维护，建议所有关键文档和基线表都使用统一状态：

- `draft`：草稿，允许频繁修改
- `review`：评审中，内容相对稳定
- `approved`：已确认，可作为当前有效版本
- `locked`：已锁口，原则上不直接改动
- `deprecated`：已废弃，不再继续维护
- `archived`：已归档，仅保留追溯

## 12. 版本关系字段建议

建议在基线表、文档元信息或变更记录中增加：

- `baseline_version`
- `doc_version`
- `supersedes`
- `archived_at`
- `doc_status`

这样可以明确：

- 当前文档基于哪个基线版本
- 当前文档替代了哪个旧版本
- 旧版本何时归档

## 13. 归档规则

建议项目实例文档在 `docs/_archive/<version>/` 下保留归档副本。

推荐规则如下：

- 小修订可以直接覆盖当前有效文档，但必须补变更记录
- 大版本切换前，先归档当前有效文档，再更新正式目录
- 已废弃文档不应继续停留在正式目录中假装有效，应移动或明确标记为废弃

稳定 ID 原则上不归档重建，而应通过 `supersedes` 或废弃标记延续追踪关系，具体规则见 `12_id_traceability_spec.md`。

## 14. 变更中的人类确认规则

如果迭代变更涉及以下内容，原则上必须经过人类确认后才能锁口：

- 范围新增或删除
- 关键页面结构调整
- 规则口径调整
- 状态定义调整
- 验收口径调整

产品 Agent 可以先完成影响分析和候选方案，但不得绕过确认直接把候选方案视为最终结论。

## 15. 变更后的评审与签收

如果本轮变更影响正式交付内容，建议重新执行：

1. 人类评审
2. 评审意见返修
3. 签收记录更新
4. 当前有效版本索引更新

相关规则见：

- `13_review_signoff_spec.md`
- `14_delivery_index_spec.md`
