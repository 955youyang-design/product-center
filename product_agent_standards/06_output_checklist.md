# 交付检查清单

| 字段 | 内容 |
|---|---|
| 文档名称 | 06_output_checklist.md |
| 文档类型 | 交付检查清单 |
| 适用范围 | 从 0 到 1 交付、版本迭代交付、文档复核 |
| 适用角色 | 产品经理、产品 Agent |
| 当前版本 | V1 |
| 维护人 | 产品 Agent 标准维护者 |
| 更新时间 | 2026-04-17 |

## 1. 从 0 到 1 交付检查

在判断一轮从 0 到 1 输出是否完成前，至少确认以下事项：

- 业务目标已经明确
- 用户、角色、端和渠道已经明确
- 当前范围与范围外边界已经明确
- 功能清单表已经形成
- 页面树表已经形成
- 功能承载图已经形成
- `docs/00-05` 分层文档已经形成
- 假设清单已经列出
- 待确认问题已经列出
- 如存在疑问或不确定项，问题清单已经整理
- 阻塞性问题已经完成人类确认，或明确标记为未过门槛

## 2. 迭代交付检查

在判断一轮迭代是否完成前，至少确认以下事项：

- 三份输入基线已经先回写
- 影响分析已经完成
- 受影响文档已经更新
- 未受影响文档没有被无故重写
- 本轮变更摘要已经给出
- 本轮新增问题已经进入问题清单
- 需要拍板的问题已经完成人类确认

## 3. 质量检查

所有文档至少要满足以下质量要求：

- 结构清晰
- 命名一致
- 上下游可追溯
- 页面与非页面节点边界清晰
- 适合产品、设计、研发、测试共同使用
- 章节结构符合统一模板骨架

## 4. 流程图检查

检查整体业务流程文档时，至少确认：

- 已包含 1 张总览流程图
- 如果存在多条业务线，已按业务线拆成多张时序图
- 时序图表达了角色、页面、机制、对象、接口之间的先后关系

## 5. 优先级检查

检查功能清单表时，至少确认：

- 优先级统一使用数字
- 优先级从 `0` 开始
- 数字越小优先级越高
- 没有混用 `P0 / P1 / P2`

## 6. 最小交付包检查

一个可交给后续团队继续推进的最小交付包，应至少包含：

1. 范围摘要
2. 假设清单
3. 待确认问题清单
4. 功能清单表
5. 页面树表
6. 功能承载图
7. 分层文档目录
8. 当前版本说明

## 7. 完整交付包检查

一套完整可落地的产品文档交付包，应至少包含：

1. 三份输入基线
2. `00-05` 全量分层文档
3. 变更记录
4. 影响分析摘要
5. 当前版本边界说明

如果页面已经进入研发实施范围，还应确认：

6. 页面开发脚手架定义已形成
7. 页面规格与脚手架定义之间可互相追溯

如果项目为中大型或多 Agent 协同，还应确认：

8. 需求追踪矩阵已形成
9. 追踪矩阵与基线、分层文档、测试用例保持一致

如果项目已进入正式交付阶段，还应确认：

10. 人类评审已完成
11. 评审意见已闭环
12. 签收记录已形成
13. 当前有效版本清单已更新

## 8. 评审结论检查

在提交给产品复核前，产品 Agent 应明确给出结论：

- 哪些内容已确认
- 哪些内容是当前假设
- 哪些问题必须在下轮确认
- 当前哪些文档可以视为稳定版本

并补充：

- 当前未关闭问题总数
- 当前阻塞问题总数
- 已完成人类确认的问题数
- 仍待确认的人类确认项

## 9. 通过标准

只有当“范围、基线、分层文档、变更记录”四类内容都能互相对齐时，才能判定本轮输出合格。
