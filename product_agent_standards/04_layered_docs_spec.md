# 分层文档规范

| 字段 | 内容 |
|---|---|
| 文档名称 | 04_layered_docs_spec.md |
| 文档类型 | 分层文档规范 |
| 适用范围 | 项目实例文档结构、命名规则、最小输出要求 |
| 适用角色 | 产品经理、产品 Agent、交互设计、研发、测试 |
| 当前版本 | V1 |
| 维护人 | 产品 Agent 标准维护者 |
| 更新时间 | 2026-04-17 |

## 1. 文档目标

本文件定义项目实例文档的目录结构、命名规则、分层职责和最小输出要求。

目标是让后续任何项目都能按同一结构稳定产出文档，而不是每次重新发明目录和章节体系。

## 2. 项目实例文档目录结构

项目实例文档建议统一放在 `docs/` 目录下，并固定为六层：

```text
docs/
  00-总览层/
  01-端产品层/
  02-专题机制层/
  03-页面设计层/
  04-数据与接口层/
  05-验收与测试层/
```

分层顺序固定为：

`总览 -> 端产品 -> 专题机制 -> 页面设计 -> 数据与接口 -> 验收与测试`

项目实例中建议额外维护一份“项目交付索引与当前有效版本清单”，作为正式交付入口，规则见 `14_delivery_index_spec.md`。

## 3. 文件命名规则

### 3.1 层级目录命名

层级目录统一使用：

`NN-中文层级名`

例如：

- `00-总览层`
- `01-端产品层`

### 3.2 层内文件命名

层内文件统一使用：

`NN-中文文档主题.md`

例如：

- `00-文档总览与目录定义.md`
- `04-整体业务流程图.md`
- `05-平台电脑端产品文档.md`

补充规则：

- 中文文档标题允许调整
- 文件路径不应因中文标题微调而频繁改名
- 建议通过 `doc_id` 或 `doc_slug` 维持稳定引用

### 3.3 页面设计层子目录命名

页面设计层建议按以下路径继续拆分：

`[端名称]/[一级功能]/[页面组]/NN-文档主题.md`

这样可以同时保持目录可读性和文档稳定性。

## 4. 每份文档必须带元信息头

每份项目实例文档顶部必须带元信息表，至少包含以下字段：

| 字段 | 要求 |
|---|---|
| 文档名称 | 人类可读的中文标题，允许调整 |
| doc_id | 稳定文档标识，不随标题变化 |
| doc_slug | 稳定路径标识，不随标题变化 |
| 文档层级 | 必须属于六层之一 |
| 文档对象 | 端 / 功能 / 页面组 / 页面 / 机制 / 对象 / 接口 / 验收 |
| 适用端 | 电脑端 / 移动端 / 小程序 / H5 / 多端 |
| 所属角色 | 哪些角色使用 |
| 所属功能 | 对应的一级或二级功能 |
| 所属页面组 | 页面或机制归属的页面组 |
| 关联机制 | 关联的专题机制 |
| 关联对象 | 关联的业务对象 |
| 上游文档 | 依赖来源 |
| 下游文档 | 继续落地的文档 |
| 当前版本 | MVP / V1 / V2 等 |
| 维护人 | 负责人 |
| 更新时间 | 最近更新时间 |

如项目已经进入长期维护阶段，建议额外增加：

- `baseline_version`
- `doc_version`
- `doc_status`
- `supersedes`

正式输出时，章节骨架建议优先遵循 `11_template_library.md`。

中文文档名称与稳定路径的关系，建议遵循 `16_navigation_naming_resilience_spec.md`。

## 5. 六层文档职责

### 5.1 00-总览层

回答：

- 产品是什么
- 解决什么问题
- 当前版本做什么、不做什么
- 主流程如何跑通
- 核心状态如何流转

最低输出建议：

- 文档总览与目录定义
- 产品总述
- 业务模型与角色协同
- 状态流转机制说明
- 整体业务流程图
- 文档治理与 Agent 要求

流程图补充要求：

- 至少 1 张总览流程图
- 如果存在多条业务线，必须按业务线拆分多张时序图
- 时序图用于表达角色、页面、机制、对象、接口之间的先后关系

### 5.2 01-端产品层

回答：

- 每个端服务谁
- 每个端承接哪些功能
- 每个功能包含哪些页面组
- 每个页面组包含哪些页面

每份端产品文档最低应包含：

1. 端定位
2. 使用角色
3. 核心场景
4. 导航结构
5. 一级功能总表
6. 功能清单表
7. 页面树表
8. 功能承载图
9. 页面组与页面映射关系
10. 跨端边界与协同关系
11. 优先级与版本范围

### 5.3 02-专题机制层

回答：

- 规则能力如何触发
- 机制如何流转
- 失败如何处理
- 页面和对象如何承接

重点承接内容：

- 规则能力
- 跨页面机制
- 跨对象机制
- 状态流转机制

### 5.4 03-页面设计层

回答：

- 页面目标
- 页面结构
- 页面布局
- 主次操作
- 字段与校验
- 状态与反馈

强制规则：

- 页面文档只写页面节点
- 功能动作和规则能力不伪装成页面
- 页面组文档负责承接跨页面动作和页面间协同

当页面进入研发实施阶段时，建议在页面设计层同步输出“页面开发脚手架定义”，用于承接页面规格到前端骨架的映射。

### 5.5 04-数据与接口层

回答：

- 核心对象有哪些
- 字段如何定义
- 状态如何统一
- 接口如何提供
- 事件与日志如何记录

最低输出建议：

- 核心对象定义
- 逻辑数据模型
- 字段字典
- 状态与枚举字典
- 接口清单
- 事件与日志清单
- 表设计建议

### 5.6 05-验收与测试层

回答：

- 如何证明需求已实现
- 主流程如何验收
- 异常流程如何验收
- 权限如何验收
- 状态流转如何验收

最低输出建议：

- 验收标准
- 主流程用例
- 异常用例
- 权限用例
- 状态流转用例
- 回归测试范围

## 6. 分层之间的上下游关系

建议按以下顺序生成与维护：

1. `00-总览层`
2. `01-端产品层`
3. `02-专题机制层`
4. `03-页面设计层`
5. `04-数据与接口层`
6. `05-验收与测试层`

上游负责定义边界，下游负责细化实现，不允许下游反向定义上游范围。

## 7. 推荐最小文件集

为了避免同一层随意拆成 1 份或 5 份，建议至少遵循以下最小文件集：

| 层级 | 推荐最小文件集 | 可合并规则 |
|---|---|---|
| 00-总览层 | 文档总览、产品总述、角色协同、状态流转、业务流程、文档治理 | 小项目可合并为 2 到 3 份，但业务流程与文档治理不建议合并 |
| 01-端产品层 | 每个端 1 份端产品文档 | 多渠道但同端同流程时可合并为 1 份，并在章节内分渠道说明 |
| 02-专题机制层 | 每个机制 1 份机制文档 | 极简单机制可并入端产品文档，但一旦跨页面或跨对象必须独立成文 |
| 03-页面设计层 | 每个页面组 1 份页面组文档；每个核心页面 1 份页面文档 | 简单弹窗、抽屉可作为主页面子章节，不强制单独成文 |
| 04-数据与接口层 | 对象定义、字段字典、状态字典、接口清单、事件日志至少各 1 份 | 小项目可合并对象与字段、接口与事件，但状态字典建议独立 |
| 05-验收与测试层 | 验收标准、测试用例、回归范围至少各 1 份 | 小项目可合并为 1 份测试文档 |

## 8. 拆文规则

当项目规模变大时，按以下优先级拆文：

1. 先按端拆
2. 再按业务线拆
3. 再按页面组拆
4. 再按页面拆
5. 再按机制拆

补充规则如下：

- 多终端时，优先“一端一文档”，不要把所有端混进同一份主文档
- 多业务线时，流程图、端产品章节和测试用例都应按业务线拆分
- 多页面组时，页面设计层优先“一页面组一文档”
- 多机制时，专题机制层优先“一机制一文档”

页面开发脚手架定义建议优先按页面组拆分；当页面差异较大时，可进一步按页面拆分。

## 9. 覆盖率回查规则

文档输出后，必须能够从基线反查到下游文档：

- `feature_id` -> 端产品层章节或映射表
- `page_group_id` -> 页面组文档
- `page_id` -> 页面文档或页面组中的页面章节
- `rule_capability` -> 专题机制文档
- `data_capability` -> 数据与接口层文档

无法回查的节点，视为未完成落文。

对于中大型项目，建议补充需求追踪矩阵，并以矩阵作为全局回查入口。追踪矩阵规则见 `12_id_traceability_spec.md`。

如果项目已进入正式交付和评审阶段，还应能从项目交付索引中直接定位当前有效版。交付索引规则见 `14_delivery_index_spec.md`。

## 10. 文档完成标准

一套合格的项目实例文档，至少要满足：

- 目录结构稳定，命名一致
- 每层回答的问题明确，不混写
- 上游基线能追溯到下游文档
- 下游文档能回溯到上游范围定义
- 后续迭代时能快速判断受影响层级

如果页面已经进入研发范围，还应满足：

- 核心页面已补充页面开发脚手架定义
- 页面文档中的主要区块、动作、状态都能映射到脚手架骨架
