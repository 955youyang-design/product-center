# 问题整理与人类确认规范

| 字段 | 内容 |
|---|---|
| 文档名称 | 10_issue_confirmation_spec.md |
| 文档类型 | 问题整理与人类确认规范 |
| 适用范围 | 基线输出、分层文档输出、迭代变更中的疑问与不确定项治理 |
| 适用角色 | 产品经理、产品 Agent、业务方、研发负责人、测试负责人 |
| 当前版本 | V1 |
| 维护人 | 产品 Agent 标准维护者 |
| 更新时间 | 2026-04-17 |

## 1. 文档目标

本文件用于约束：在文档阶段出现疑问、不确定项、分歧或多种合理解释时，产品 Agent 应如何整理问题、如何请求人类确认，以及在确认前后如何回写文档。

## 2. 核心原则

### 2.1 不静默处理

只要存在疑问或不确定项，产品 Agent 不得在没有记录的情况下直接拍板。

### 2.2 不用默认处理替代确认

默认处理只是临时推进方式，不是最终结论。

### 2.3 不把未确认问题伪装成已确认规则

文档中任何带假设推进的内容，都必须可追溯到问题清单。

## 3. 什么情况必须进入问题清单

以下情况必须进入问题清单：

- 当前版本范围存在歧义
- 页面结构存在两种以上合理拆法
- 规则能力、状态流转或对象口径不一致
- 接口依赖、外部系统依赖尚未明确
- 验收标准存在多种解释
- 不同 Agent、不同角色之间出现理解冲突

## 4. 问题分级建议

建议至少分为四级：

| 阻塞等级 | 含义 | 处理要求 |
|---|---|---|
| `L0-阻塞基线` | 不确认就无法进入三份输入基线 | 必须先做人工确认 |
| `L1-阻塞分层文档` | 可建基线，但不能进入正式文档交付 | 原则上先确认再输出正式稿 |
| `L2-阻塞研发实施` | 文档可继续，但研发不可按此实施 | 可先写草稿，交付前必须确认 |
| `L3-不阻塞当前推进` | 可带假设继续推进 | 必须记录默认处理方式和确认责任人 |

## 5. 问题清单模板

建议使用如下字段：

| issue_id | 问题类型 | 问题内容 | 影响范围 | 阻塞等级 | 默认处理方式 | 待确认角色 | 待确认人 | 当前状态 | 备注 |
|---|---|---|---|---|---|---|---|---|---|

字段说明建议如下：

- `问题类型`：范围 / 页面 / 机制 / 对象 / 接口 / 测试 / 术语
- `影响范围`：写受影响的文档层级、页面组、对象或机制
- `当前状态`：待确认 / 已确认 / 已关闭 / 已废弃

## 6. 人类确认记录模板

建议使用如下字段：

| issue_id | 确认方式 | 确认结论 | 确认人 | 确认角色 | 确认时间 | 影响文档 | 是否已回写 | 关闭状态 |
|---|---|---|---|---|---|---|---|---|

`确认方式` 建议至少支持：

- 口头确认
- 文档批注确认
- 会议确认
- 消息确认

## 7. 必须发起人类确认的场景

以下场景原则上必须发起人类确认：

- 新增或删除当前版本范围
- 页面树结构发生关键变化
- 规则能力或状态口径变更
- 关键对象字段定义变更
- 验收通过标准变化
- 多个候选方案都会影响研发与测试

## 8. 默认处理规则

在确认前允许的默认处理必须满足：

- 不扩大当前已确认范围
- 不创建新的主业务线
- 不改变已锁定的关键术语和状态口径
- 在问题清单中有明确记录

如果不满足以上条件，不允许使用默认处理继续推进。

## 9. 回写规则

人类确认完成后，产品 Agent 必须做两件事：

1. 更新问题清单和确认记录
2. 回写受影响的基线表和分层文档

如果只完成确认但未回写文档，问题不能视为真正关闭。

## 10. 交付门槛

在正式交付前，至少应满足：

- 所有 `L0-阻塞基线` 问题已确认
- 所有 `L1-阻塞分层文档` 问题已确认
- 所有 `L2-阻塞研发实施` 问题已确认，或明确禁止进入研发
- 所有 `L3-不阻塞当前推进` 问题都已标记默认处理方式和确认责任人

## 11. 建议输出位置

问题清单与确认记录建议放在：

`docs/00-总览层/06-问题清单与确认记录.md`

如果项目较大，也可以拆成：

- `06-问题清单.md`
- `07-确认记录.md`

## 12. 完成标准

一套合格的问题整理与人类确认机制，至少应满足：

- 任何关键不确定项都可被追溯
- 任何阻塞问题都能看到确认责任人和确认结论
- 任何确认结论都已回写到对应文档
- 产品 Agent 没有用默认处理替代正式拍板
