## 1. 文档治理目标

- 保证当前项目的功能范围、页面范围、机制、数据、测试始终围绕同一套基线推进。
- 让后续产品 Agent 接手时，能沿同一文档结构继续补写和维护，而不是重建体系。
- 控制中文文档名、菜单名、页面名经常变动时的稳定引用问题。

## 2. 基本治理原则

1. 基线优先：任何范围变化先回写 `docs/baselines/`，再回写 00-05 层文档。
2. 稳定标识优先：`feature_id`、`page_id`、`page_group_id`、`route_key`、`menu_key`、`doc_id` 必须优先保持稳定。
3. 中文名称可调：中文文档标题、中文菜单名称、中文页面名称可以调整，但不得因为名称微调频繁改稳定标识。
4. V1.0和V1.0分开写：V1.0文档可以保留，但不得写成当前已交付。
5. 历史兼容单独标识：历史路径允许保留，但必须明确 `current-effective`，不得混入当前交付链路。

## 3. 变更回写顺序

当产品范围发生变化时，必须按以下顺序回写：

1. `docs/baselines/00-feature-list.md`
2. `docs/baselines/01-page-tree.md`
3. `docs/baselines/02-functional-carrying-diagram.md`
4. `docs/00-总览层/`
5. `docs/01-端产品层/`
6. `docs/02-专题机制层/`
7. `docs/03-页面设计层/`
8. `docs/04-数据与接口层/`
9. `docs/05-验收与测试层/`
10. `docs/00-总览层/07-项目交付索引与当前有效版本清单.md`

## 4. 问题确认要求

- 文档阶段遇到不确定项时，必须先登记到 `docs/00-总览层/06-问题清单与确认记录.md`。
- 每个问题至少写清：问题内容、影响范围、阻塞等级、默认处理方式、待确认角色、当前状态。
- 如果用户已经在对话中确认，必须把确认结论回写到确认记录表，并同步更新受影响文档。
- 未确认但不阻塞推进的问题，可以按默认口径继续，但必须保留记录。
- 阻塞研发实施的问题，必须在 `09-评审签收与交付门槛.md` 中体现。

## 5. Agent 工作要求

- 新任务开始前，先读 `docs/baselines/` 和 `07-项目交付索引与当前有效版本清单.md`。
- 续写或改写文档时，只在 `docs/` 规范路径下落文档。
- 新增文档必须同步进入交付索引。
- 修改已有文档时，必须同步更新 `doc_version`、`更新时间`，必要时更新 `doc_status`。
- 页面、机制、数据、测试之间出现追踪断链时，优先修正 `08-需求追踪矩阵.md`。
- 涉及页面开发映射时，必须同步检查 `03-页面设计层/中台用户端/99-页面开发脚手架定义.md`。

## 6. 命名与路径要求

- 文档文件名统一使用 `NN-中文主题.md`。
- 页面设计层按 `中台用户端/一级功能/页面组/NN-文档主题.md` 组织。
- 中文文档名可以调整，但路径变更必须少做，避免破坏引用链。
- 如确实需要改路径，必须同步回写：
  - `00-文档总览与目录定义.md`
  - `07-项目交付索引与当前有效版本清单.md`
  - `08-需求追踪矩阵.md`
  - 受影响上游/下游文档

## 7. 交付检查要求

每次文档阶段交付前，至少检查以下项目：

- 基线三件套是否一致
- V1.0页面是否都有页面文档
- V1.0机制是否有专题机制文档
- V1.0对象/接口/测试是否能追踪到
- 当前阻塞问题是否已登记
- 交付索引是否包含本轮新增或改写文档

## 8. 当前项目特别约束

- 当前V1.0支持淘宝、拼多多、抖音、快手四渠道导入。
- 商品管理承接导入后的全渠道商品统一管理和编辑，不限定单一渠道。
- 历史独立发布相关页面仅保留占位说明，不纳入当前交付链路。
