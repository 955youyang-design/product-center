## 4. 对开发的菜单路由调整建议

| 调整对象 | 文档口径 | 开发动作 |
|---|---|---|
| 一级导航容器 | 顶层顺序固定为产品、商品、库存、配置、素材 | 以 `nav_order=10/20/30/40/50` 回写 `product-platform.ts` 顶层容器顺序。 |
| 顶层 redirect | 每个一级菜单容器都应 redirect 到当前默认叶子页 | 保持 `/products -> /products/list`、`/channel-items -> /channel-items/list`、`/inventory -> /inventory/list`、`/store-config -> /store-config/list`、`/materials -> /materials/list`。 |
| 商品管理拆页 | `渠道商品详情` 只管单商品售卖字段；`渠道商品详情页列表` 统一管理全部商品详情页内容、版本和素材复用 | 新增 `/channel-items/detail-pages` 二级菜单路由，并让详情页数量/关联摘要统一跳该页；该页内需支持编辑工作区、当前版本切换和已有素材引用，不要继续把详情页内容编辑塞回 `/channel-items/detail`。 |
| 类目动态适配 | 店铺级类目动态适配归中台管理端类目映射和属性值映射维护 | 用户端商品详情只调用 `API-ITEM-CATEGORY-ADAPTATION-GET` 读取结果；不要在用户端配置管理或商品详情页写映射规则。 |
| 中台管理端根路由 | 中台管理端是独立端，推荐根路径 `/management` | 建立独立菜单树：客户管理、类目映射、属性值映射、系统管理，不挂入中台用户端菜单。 |
| 详情 / 记录页 | 详情页、记录页的隐藏路由策略属于实现层 | 可以使用 `hideInMenu` + `activePath`，但不得把这种实现策略反写成上游入口事实。 |
| 二级入口 | 产品导入、渠道商品详情页列表为V1.0二级菜单；批量修改、库存调整记录为V1.0二级入口 | `menu.product.import`、`menu.channel_item.detail_page_list` 当前可见；`menu.channel_item.batch_edit`、`menu.inventory.adjust_log` 按 V1.0 同级入口补齐。 |
| 详情页管理约束 | 一个产品会映射多个渠道商品；单个渠道商品最多 3 个详情页 | 列表页要支持按产品聚合查看多个渠道商品的详情页，并在新建/上传/复制动作上直接做 3 条上限校验。 |
| 素材复用 | 同一图片允许进入多个商品详情页 | 在详情页列表中提供素材选择/引用能力；素材原子维护仍归 `menu.asset`。 |
| V1.0顶层菜单 | `menu.asset` 作为中台用户端一级菜单 | 不改 `page_id`、`menu_key`、`route_key`。 |
| V1.0动作路由 | 新增产品与批量修改的入口语义不同 | 新增产品维持“产品列表按钮进入”；批量修改如要独立路由，应挂在商品管理二级菜单，不要改写成列表按钮入口。 |
| 历史兼容目录 | `overview/`、`system-config/`、`publish/`、`operations/`、`reports/` 不属于当前菜单树 | 开发不要再从这些目录反推当前路由；如要复用，先回写 baselines。 |
